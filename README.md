# 🚍 Ticket Service API 

## 📌 Описание проекта

Ticket Service — это REST API, разработанное на **Java (Spring Framework)** для управления транспортными билетами.
Система реализует логику, при которой:

* **Пользователь** может зарегистрироваться, авторизоваться и покупать билеты.
* **Компания-перевозчик (Carrier)** создается пользователем. Один пользователь может владеть **только одной компанией**.
* **Маршрут (Route)** создается перевозчиком.
* **Билеты (Tickets)** привязываются к маршруту.

Пользовательский сценарий:

1. Зарегистрироваться в системе.
2. Создать **компанию (Carrier)**.
3. Создать **маршрут (Route)**.
4. Создать **билеты (Tickets)** для маршрута.
5. Другие пользователи могут покупать и возвращать билеты.

---

## ⚙️ Используемые технологии

* **Spring Boot** — каркас приложения
* **Spring Security + JWT (Nimbus-JWT)** — аутентификация и авторизация
* **JdbcTemplate** — работа с базой данных
* **PostgreSQL** — СУБД
* **Redis** — хранение активных токенов, сессий, logout
* **Flyway** — миграции базы данных
* **Swagger/OpenAPI** — документация API
* **MapStruct** — маппинг DTO ↔ Entity
* **Lombok** — сокращение шаблонного кода
* **Docker** — для контейнеризации

---

## 🔐 Авторизация и безопасность

* Используется **JWT** с разделением на:

    * **Access Token** (жизнь: 15 минут)
    * **Refresh Token** (жизнь: 30 дней)
* Собственные **JWT-фильтры**:

    * Если пользователь не авторизован → `401 Unauthorized`.
* Поддерживается:

    * Получение информации о всех токенах пользователя.
    * **Logout** с инвалидированием токенов через Redis.
    * Автоматический выход со всех устройств при смене пароля.
    * При удалении пользователя все JWT-токены автоматически удаляются.

---

## 👤 API пользователя

* Регистрация, авторизация, обновление, удаление, восстановление.
* Получение **списка купленных билетов** (с пагинацией + кэшированием).
* Получение **всех активных токенов пользователя**.

---

## 🏢 API компании (Carrier)

* Создание, обновление, удаление, восстановление.
* Каждая компания имеет **уникальное название**.
* После удаления название резервируется за пользователем на **30 дней**.
* По истечении 30 дней название доступно другим пользователям.

---

## 🛣 API маршрутов (Route)

* Создание, удаление.
* При удалении маршрута автоматически удаляются **все связанные билеты**.

---

## 🎟 API билетов (Tickets)

* Создание (привязка к маршруту).
* Покупка билета пользователем.
* Возврат билета.

---

## 📊 Бизнес-логика (схема)

Вот как выглядит процесс:

```
[ Пользователь ]
       │
       ▼
[ Создать перевозчика (Carrier) ]
       │
       ▼
[ Создать маршрут (Route) ]
       │
       ▼
[ Создать билеты (Tickets) ]
       │
       ▼
[ Другие пользователи → Купить/Вернуть билеты ]
```
